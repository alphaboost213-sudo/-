# ImgUniq — уникализатор изображений

Сервис принимает ссылку на изображение и выдаёт статическую ссылку вида:

```text
https://твой-сервис.railway.app/img/abc123
```

При каждом открытии этой ссылки сервис заново скачивает исходник, применяет случайные микротрансформации и отдаёт новый JPEG-файл.

## Режимы исходной ссылки

Теперь есть два режима:

### Обычная исходная ссылка

Используется по умолчанию. Сервис хранит ссылку в SQLite и при каждом открытии `/img/<id>` заново скачивает исходник.

### Динамическая исходная ссылка

Включается чекбоксом **«Динамическая исходная ссылка»** в интерфейсе или параметром API:

```json
{"source_type":"dynamic_url"}
```

Этот режим нужен, если исходная ссылка сама каждый раз генерирует новое изображение, например:

```text
https://example.com/render?id=123
```

Для таких ссылок ImgUniq при каждом запросе добавляет cache-buster:

```text
_imguniq_cb=<timestamp>_<random>
```

То есть исходник запрашивается не как кешированная картинка, а как новый рендер.

## Что делает уникализация

При каждом рендере применяется набор лёгких случайных изменений:

- случайный шум;
- низкоамплитудная текстура;
- микросдвиг яркости, контраста и насыщенности;
- лёгкий blur или sharpening;
- микрокроп 0–2 px с ресайзом обратно;
- случайный ресемплинг через промежуточный размер;
- субпиксельный affine-сдвиг;
- JPEG re-encode с плавающим `quality`;
- случайные параметры JPEG `progressive` и `subsampling`;
- новый EXIF-блок при каждом сохранении.

Визуально изображение почти не меняется, но технически файл каждый раз другой.

## Хранение ссылок

Ссылки больше не хранятся в RAM. Используется SQLite:

```text
imguniq.sqlite3
```

Для Railway лучше подключить Volume и задать переменную окружения:

```text
DATA_DIR=/data
```

Тогда база будет лежать здесь:

```text
/data/imguniq.sqlite3
```

Без Volume база может пропасть при пересборке или переносе окружения Railway.

## Как деплоить на Railway

1. Загрузи файлы проекта в GitHub.
2. Railway → New Project → Deploy from GitHub repo.
3. Подключи Volume.
4. Mount path: `/data`.
5. Variables → добавь `DATA_DIR=/data`.
6. Deploy / Redeploy.

В `railway.toml` и `Procfile` стоит один worker:

```text
gunicorn main:app --bind 0.0.0.0:$PORT --workers 1 --timeout 120
```

Так стабильнее для SQLite и обработки изображений на небольшом Railway-сервисе.

## API

### Создать обычную ссылку

```http
POST /api/register
Content-Type: application/json

{"url":"https://example.com/image.jpg"}
```

### Создать динамическую ссылку

```http
POST /api/register
Content-Type: application/json

{"url":"https://example.com/render?id=123", "source_type":"dynamic_url"}
```

### Создать пачку динамических ссылок

```http
POST /api/register-batch
Content-Type: application/json

{
  "urls":[
    "https://example.com/render?id=1",
    "https://example.com/render?id=2"
  ],
  "source_type":"dynamic_url"
}
```

Лимит пачки — 50 ссылок за раз.

### Получить статистику

```http
GET /api/stats
```

Ответ содержит:

```json
{
  "total_registered": 10,
  "dynamic_registered": 3,
  "storage": "/data/imguniq.sqlite3",
  "uptime": "ok"
}
```

## Важно

- SQLite хранит только исходные ссылки, не сами изображения.
- Старые записи автоматически считаются обычными ссылками `static_url`.
- Для `dynamic_url` сервис добавляет `_imguniq_cb`, чтобы источник не отдавал кеш.
- Если исходная ссылка перестанет работать, уникализированная ссылка тоже перестанет отдавать изображение.
- Для полного автономного хранения картинок нужно отдельно добавлять скачивание оригиналов в `/data/uploads`.
